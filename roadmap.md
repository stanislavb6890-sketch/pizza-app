## Pizza Platform — Implementation Roadmap (для ИИ)

Цель: поэтапно реализовать платформу по архитектуре из `pizza_architecture_v6.1.md`, с чёткими шагами и проверками “готово/не готово”.

> **Правило:** каждый шаг помечается как `DONE` только если все критерии проверки выполнены автоматически (тест, линтер, проверка файла и т.п.) или явно подтверждены пользователем.

---

## Фаза 0 — Базовая подготовка репозитория

### Шаг 0.1 — Создать репозиторий и базовую структуру

- **Действия:**
  - Если репозиторий ещё не инициализирован — инициализировать `git` (опционально по запросу пользователя).
  - Создать базовую структуру папок:
    - `app/`, `components/`, `modules/`, `core/`, `db/`, `services/`, `infra/`, `tests/`.
  - Создать `README.md` с кратким описанием проекта и ссылкой на `pizza_architecture_v6.1.md`.
- **Проверка (DONE, если):**
  - Все указанные директории существуют.
  - Файлы `README.md` и `pizza_architecture_v6.1.md` присутствуют в корне проекта.

### Шаг 0.2 — Инициализация Next.js проекта

- **Действия:**
  - Создать Next.js App Router проект (через `create-next-app` или вручную).
  - Убедиться, что:
    - Включён TypeScript.
    - Структура использует `app/` (App Router).
  - Настроить базовый `tsconfig.json`, `next.config.mjs`, `package.json`.
- **Проверка (DONE, если):**
  - Проект успешного собирается: `npm run build` возвращает код 0.
  - Команда `npm run dev` поднимает дев‑сервер без ошибок.

### Шаг 0.3 — Базовая конфигурация линтера и форматтера

- **Действия:**
  - Добавить ESLint + Prettier конфигурации.
  - Добавить скрипты:
    - `"lint"`, `"format"`.
- **Проверка (DONE, если):**
  - `npm run lint` проходит без ошибок (или с минимальными, согласованными с пользователем).
  - `npm run format` форматирует код без сбоев.

## Фаза 1 — БД и Prisma

### Шаг 1.1 — Настройка Prisma и подключения к MySQL

- **Действия:**
  - Установить Prisma и драйвер MySQL.
  - Создать `prisma/schema.prisma`.
  - Конфиг `DATABASE_URL` (в `.env` или через переменные среды).
- **Проверка (DONE, если):**
  - `npx prisma validate` проходит успешно.
  - `npx prisma migrate dev --name init` создаёт БД и таблицы без ошибок.

### Шаг 1.2 — Смоделировать основные сущности по архитектуре

- **Действия:**
  - В `schema.prisma` описать:
    - `User`, `AdminUser`, `Product`, `Category`, `SubCategory`,
      `Cart`, `Order`, `OrderItem`, `Address`, `DeliveryZone`, `Payment`.
  - Добавить:
    - UUID‑primary key для всех основных сущностей.
    - Поля аудита (`createdAt`, `updatedAt` и т.п.).
    - Soft‑delete поля там, где нужно (`deletedAt` или `isDeleted`).
  - Настроить индексы и внешние ключи.
- **Проверка (DONE, если):**
  - `npx prisma validate` успешен.
  - Миграции применяются без ошибок.
  - В БД реально существуют все таблицы и связи.

## Фаза 2 — Базовый каркас слоёв (Clean Architecture + DDD)

### Шаг 2.1 — Структура модулей (`modules/*`)

- **Действия:**
  - Для каждого доменного контекста (`cart`, `order`, `product`, `user`, `delivery`) создать поддиректории:
    - `domain/` (entities, value objects, доменные события).
    - `application/` (use‑cases, application services).
    - `infrastructure/` (репозитории, адаптеры).
  - Внутри каждого модуля создать хотя бы по одному заглушечному файлу.
- **Проверка (DONE, если):**
  - Все нужные директории и базовые файлы существуют.
  - Нет циклических импортов между `domain` → `application` → `infrastructure` (по правилам чистой архитектуры).

### Шаг 2.2 — Общий `core/` слой

- **Действия:**
  - Создать в `core/`:
    - `auth/` (интерфейсы для auth, типы токенов).
    - `config/` (загрузка конфигурации из env).
    - `logger/` (абстракция логгера).
    - `validation/` (DTO‑валидация, схемы).
- **Проверка (DONE, если):**
  - Во всех местах используется импорт из `core/*` как из “общей библиотеки”, а не дубликаты логики по модулям.
  - Сборка и линтер проходят.

- **Действия:**

  - E2E/интеграционные тесты подтверждают:
    - доступ к админ‑маршрутам без прав запрещён,
    - пользовательские маршруты доступны с корректной ролью.

## Фаза 4 — Каталог продуктов и корзина

### Шаг 4.1 — Домен `product`

- **Действия:**
  - Реализовать доменные сущности и репозитории для `Product`, `Category`, `SubCategory`.
  - Реализовать use‑cases:
    - Получение списка продуктов.
    - Получение продукта по id/slug.
    - Получение категорий/подкатегорий.
- **Проверка (DONE, если):**
  - Есть unit‑тесты на доменные правила (например, цена, наличие).
  - Интеграционные тесты на репозиторий чтения продуктов.

### Шаг 4.2 — Домен `cart`

- **Действия:**
  - Хранить корзину в Redis (по `userId` или `sessionId`).
  - Реализовать операции:
    - добавление товара в корзину,
    - изменение количества,
    - удаление,
    - очистка корзины.
  - Добавить валидацию по наличию товара и минимальным/максимальным значениям.
- **Проверка (DONE, если):**
  - Unit‑тесты для всех операций корзины.
  - Интеграционные тесты с Redis (чтение/запись/TTL).

### Шаг 4.3 — UI для каталога и корзины

- **Действия:**
  - Страница списка продуктов (SSR/RSC).
  - Компонент карточки продукта.
  - UI корзины с синхронизацией с сервером (через API/Route handlers).
- **Проверка (DONE, если):**
  - E2E‑тест: пользователь может добавить товар в корзину, увидеть его там, изменить количество, удалить.

## Как использовать этот roadmap ИИ‑агентом

- Всегда работать **по фазам и шагам последовательно**, не перепрыгивая через невыполненные пункты без явного запроса пользователя.
- Перед началом работы над шагом — **коротко описать пользователю**, что будет сделано.
- После выполнения — **пройтись по критериям проверки** и явно отметить, выполнены ли они.
- Если критерий не может быть полностью проверен автоматически — запросить у пользователя подтверждение или дополнительные данные.
